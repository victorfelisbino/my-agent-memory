This playbook translates attention and adoption research into practical framework messaging.

## Purpose

- Make value understandable in seconds.
- Reduce early drop-off from vague positioning.
- Guide users to a single first action.

## Core positioning model

Use this sequence on homepage, README top, and launch posts:

1. Problem in one line
- "Teams lose context faster than they lose talent."

2. Mechanism in one line
- "This framework captures decisions, turns them into verified guardrails, and keeps them current."

3. Outcome in one line
- "Fewer repeated mistakes, faster recovery, better Copilot answers."

4. First action
- "Run the quick restart routine and test one real task."

## Messaging principles (high attention retention)

1. Specific over abstract
- Prefer "recover context in under 2 minutes" over "improve productivity."

2. Recognition over recall
- Use clear labels and obvious paths so users do not guess where to click next.

3. Progressive disclosure
- Show essentials first; expose advanced controls later.

4. Information scent
- Every link label should predict the destination outcome.

5. Immediate credibility
- Pair claims with one measurable signal or operating metric.

## Homepage content pattern

1. Hero
- One sentence problem + one sentence mechanism + one clear action.

2. "Should I use this?"
- Qualification checklist with explicit "not for" criteria.

3. Quick path
- 4-step first run sequence with expected time.

4. Proof loop
- Show weekly review signals and freshness/sunset rules.

## Copy blocks you can reuse

### Short value proposition

"my-agent-memory is a shared operating framework that converts daily work into reusable, verified rules so teams recover context faster and repeat fewer mistakes."

### Clarity CTA

"Start with one real task. Run quick restart. Then decide based on outcomes, not hype."

### Not-for statement

"Not a generic notes app. Not a dump of raw transcripts. This is a governed memory system with review cadence."

## Guardrails

!!! warning "Hard messaging rules"
    - Do not use hype language without an operational definition.
    - Do not add new promise lines without a measurable success signal.
    - Keep top-level navigation to outcomes, not internal jargon.
