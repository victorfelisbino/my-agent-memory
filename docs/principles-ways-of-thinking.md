!!! note "How to read this page"
	Every item below is intentionally project-agnostic. If you find yourself wanting to add a story, client, or product name, capture it in your private notes instead and only promote the underlying pattern here.

## 01 — Core thinking principles

- **Clarity before speed.** Define the problem precisely before choosing tools.
- **Evidence over confidence.** Confidence without evidence is noise.
- **Root cause over symptom.** Avoid patching outcomes when causes are unknown.
- **Reversible first moves.** Prefer low-risk steps when uncertainty is high.
- **Leverage over effort.** Prioritize actions that unlock many downstream wins.
- **Simplicity under pressure.** Reduce moving parts when systems are fragile.

## 02 — Reasoning loop

1. **Frame.** What are we actually trying to improve?
2. **Constraints.** What cannot be broken?
3. **Options.** What are 2–3 viable paths?
4. **Tradeoffs.** Speed, risk, reversibility, complexity.
5. **Decision.** Choose one and define explicit success checks.

**Quality bar for any plan**

- Distinguish facts, assumptions, and unknowns.
- State why this path is the default.
- End every plan with verification criteria.

## 03 — Decision hygiene

**Decision card**

- Goal
- Decision type — technical, process, communication, or priority
- Time horizon — now, this week, this quarter
- Reversibility — high, medium, low
- Risk if wrong
- Cost to test quickly

**Option scoring (1–5 each)**

Impact + Confidence + Reversibility + Learning value + Alignment with goals.

**Rules**

- If reversibility is low, gather more evidence first.
- If confidence is low, run a small test before full rollout.
- If two options are close, prefer the one with faster feedback.

**Post-decision loop**

- What happened?
- What did we learn?
- What guardrail should be added?

## 04 — Bias checks

**Common traps**

- **Confirmation bias** — only looking for evidence that supports current belief.
- **Recency bias** — overweighting the latest incident.
- **Authority bias** — accepting claims without verification because the source sounds strong.
- **Action bias** — doing more instead of thinking better.
- **Sunk cost fallacy** — continuing because effort has already been invested.

**Debiasing prompts**

- What evidence would prove me wrong?
- What is the strongest alternative explanation?
- If I started fresh, would I choose the same path?
- What would I advise someone else to do here?
- Which assumption is least tested?

**Pre-execution hygiene**

- One sentence for "why now".
- One sentence for "what could fail".
- A stop condition defined before execution starts.

## What does not belong on this page

- Specific clients, employers, customers, vendors, or product names.
- Project codenames, ticket IDs, or environment URLs.
- Transcripts, raw chat logs, or anything time-bound to a single engagement.
- Anything that cannot be applied by a different person on a different project tomorrow.

If a lesson fails that bar, keep it in your private notes and promote only the generalized pattern here.
