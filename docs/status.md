# Status

**As of:** May 26, 2026

This page exists so the public-facing claims in the rest of the docs can't drift. If a feature is described as working anywhere on the site, it should appear in the **Real today** column below. If it doesn't, the docs are wrong and need fixing.

Three columns:

- **Real today** &mdash; ships end-to-end, runs on my machine right now, can be reproduced from this repo.
- **Documented only** &mdash; the pattern is written down, sometimes with example scripts, but it's not wired into a one-command path you can rely on. Treat as a design sketch.
- **Planned** &mdash; called out in the [roadmap](roadmap.md), not started.

## Real today

- **Two-repo split.** Framework lives here; private state lives in a sibling `my-agent-memory-personal` repo. Path resolution via `AGENT_MEMORY_PERSONAL`, sibling directory, or fallback.
- **Cross-machine sync.** `sync-memory.ps1` / `.sh` pulls, captures from local Copilot transcripts (with machine + workspace attribution), regenerates `active-threads.md`, optionally commits + pushes the personal repo. `.gitattributes` merge=union prevents conflicts on `observations.jsonl`.
- **One-verb capture.** `loop.ps1` / `.sh` with `idea | start | promise | wait | done | show`. Every capture also appends to `observations.jsonl`.
- **Context brief generation.** `summon-memory.ps1` / `.sh` produces a ranked brief (full or `-Compact`) with router-hints headers. Scoring is keyword-based (multi-dimension: keyword relevance + file freshness + confidence metadata + verification-date). No semantic/embedding search &mdash; relevance depends on keyword overlap between the task description and file contents.
- **Weekly synthesis loop.** `run-weekly-memory.ps1` / `.sh` orchestrates a 6-step pipeline: pull, run learner (frequency counter on Copilot transcripts &mdash; keyword trigger matching, not ML), cross-machine sync, synthesize observations, prune old entries, lint team-memory, and commit both repos.
- **Daily scheduled task install (Windows).** `install-scheduled-task.ps1` registers the daily sync. No `cron` or `launchd` equivalent is shipped (launchd logs are gitignored for local use).
- **Documented principles, gotchas, and domain playbooks** &mdash; `thinking-principles.md`, `decision-framework.md`, `cognitive-bias-checks.md`, `gotchas.md`, `salesforce-debugging.md`, `domains/`. These are the things `summon-memory` ranks and pulls from.
- **Anti-hallucination skill (load-it-yourself).** Packaged from `anti-hallucination-protocol.md` into [`skills/general/anti-hallucination/`](https://github.com/victorfelisbino/my-agent-memory/blob/main/skills/general/anti-hallucination/) with copy-paste block, per-agent install paths (Copilot / Cline / Cursor), a five-prompt before/after test harness, a results template, and a redacted [example results file](https://github.com/victorfelisbino/my-agent-memory/blob/main/skills/general/anti-hallucination/results-examples/example-redacted.md). See [Anti-hallucination skill](anti-hallucination-skill.md). Effectiveness in any specific setup still requires running the harness; real-run results across agents not yet aggregated.
- **Public probes (Wave 2).** Two upstream PRs live:
    - Probe A: Anti-hallucination skill → [`groupzer0/vs-code-agents` PR #10](https://github.com/groupzer0/vs-code-agents/pull/10), awaiting review.
    - Probe B: Quality gate middleware (full TS port of 40+ scoring rules) → [`alioshr/memory-bank-mcp` PR #35](https://github.com/alioshr/memory-bank-mcp/pull/35), awaiting review.
- **Competence map.** [`competence-map.yml`](https://github.com/victorfelisbino/my-agent-memory/blob/main/competence-map.yml) at the repo root + [`scripts/generate-competence-map.ps1`](https://github.com/victorfelisbino/my-agent-memory/blob/main/scripts/generate-competence-map.ps1) generator + [Competence map](competence-map.md) page. Depth is author-set; the generator refuses to render `expert` claims without backing evidence and downgrades anything untouched for 90+ days to dormant. Eight shared-scope entries seeded. CI drift-check job rejects PRs that change the YAML without regenerating the page; weekly script regenerates on every run. Two-repo merge with the sibling personal repo + Actions cron are roadmap follow-ups.
- **Admission-gate measurement harness + live write-path gate (Wave 3, v4 + v5 + v6 fixtures + 14 iterations, both PowerShell and Python scorers with cross-language parity, local HTML dashboard, and capture-time gating on every write path).** [`admission-gate/`](https://github.com/victorfelisbino/my-agent-memory/blob/main/admission-gate/) holds a 100-item labeled fixture (50 keep, 50 reject), an iter-10 v5 extension (108 items: v4 + 4 contradicts-store + 4 reinforces-store), an iter-11 v6 extension (116 items: v5 + 4 distinct-topic keeps + 4 feedback-loop rejects), a 6-anchor store fixture, a 4-item recalled-session fixture, a corpus extractor, a baseline stub scorer in both [`score-memory.ps1`](https://github.com/victorfelisbino/my-agent-memory/blob/main/admission-gate/score-memory.ps1) and [`score_memory.py`](https://github.com/victorfelisbino/my-agent-memory/blob/main/admission-gate/score_memory.py), and an iter-12 dashboard renderer [`render-dashboard.ps1`](https://github.com/victorfelisbino/my-agent-memory/blob/main/admission-gate/render-dashboard.ps1) that reads the shared scoring log and emits a single self-contained `dashboard.html` (gitignored, local-only). Current baseline: **100/100/100 on v4 (no store)**, **108/108/108 on v5 (with `-Store`)**, and **116/116/116 on v6 (with `-Store` + `-Recalled`)** in both languages. Iter 7 cleared the Wave 3 exit criterion for the first time at 91/100/82; iter 8 closed the nine named misses and saturated v4 at 100/100/100; iter 9 lifted the rules into a standalone Python port so the scorer can be embedded as middleware outside this repo's PowerShell-native CI; iter 10 wired the previously-stubbed novelty dimension to a contradiction-against-store check (polarity + subject-overlap; -2.0 when same subject + opposite polarity vs any anchor); iter 11 extended novelty with a feedback-loop check (-2.0 when same subject + same polarity AND >= 4 shared tokens vs any recalled-session item; higher overlap bar than contradiction because same polarity needs more evidence to distinguish redundancy from topic similarity); iter 12 added a per-item JSON log (`-LogTo` / `--log-to`) and a static HTML dashboard so decisions are reviewable -- no server, no JS framework, no rule changes; iter 13 wired the scorer into the manual write path via a new `-ScoreOne` / `--score-one` flag (stdin JSON in, decision JSON out, exit 0 keep / 3 reject) consumed by [`capture-observation.ps1`](https://github.com/victorfelisbino/my-agent-memory/blob/main/capture-observation.ps1) and `capture-observation.sh`, with rejects diverted to `observations.rejected.jsonl` and bypass via `-NoGate` / `--no-gate` or `MEMORY_GATE=off`; iter 14 closed the auto-capture bypass by wiring the same per-candidate gate into [`auto-capture-observations.ps1`](https://github.com/victorfelisbino/my-agent-memory/blob/main/auto-capture-observations.ps1) and `auto-capture-observations.sh` -- the scripts that run weekly via `run-weekly-memory.ps1` and on every sync via `sync-memory.ps1` (the volume path) -- so the gate now filters every write to `observations.jsonl`, not just manual ones. Cross-language parity is enforced in CI by [`parity-check.ps1`](https://github.com/victorfelisbino/my-agent-memory/blob/main/admission-gate/parity-check.ps1): same fixture in, same per-item keep/reject out, with or without `-Store` / `-Recalled`. Today CI verifies parity on v4 (100 items), v5 + store (108 items), v6 + store + recalled (116 items), the live real-memory corpus (381 items, no store), the real-memory corpus with store, and the real-memory corpus with store + recalled; a one-sided rule change will fail the build. CI also smoke-tests the dashboard end-to-end (PS log + Py log + render + assert expected sections present), the manual write-path end-to-end (PS + Py `--score-one` exit codes plus a full keep/reject/bypass run through `capture-observation.ps1` verifying the resulting files contain exactly the expected lines), and the auto-capture write-path end-to-end (synthetic transcript fed through `auto-capture-observations.ps1` gated + bypass, plus `auto-capture-observations.sh` gated, asserting kept/rejected counts). Real-corpus rejection is 1.0% (4 of 381 items: 1 Status snapshot + 3 personal-goal TODO bullets, all defensible), unchanged by the iter-10 store check and iter-11 feedback-loop check (no real bullet collides with the recalled set).
- **CI hardening.** Three jobs: recursive `*.ps1`/`*.sh` parse, `mkdocs build --strict`, competence-map drift check, admission-gate harness run. All required to pass before merge in practice (branch-protection wiring is admin-bypass for now).
- **mkdocs site.** Builds clean with `--strict`, deploys via GitHub Pages workflow.

## Documented only (NOT yet shipped end-to-end)

- **Auto-injected anti-hallucination protocol.** The skill itself now ships as load-it-yourself (see Real today). Auto-injection by `summon-memory` is still not implemented; that's [roadmap](roadmap.md) Wave 4-A territory (MCP server).
- **Transfer-test promotion gate.** The rule ("only promote if it would still apply in a language you haven't met yet") is written into [framework-scope.md](framework-scope.md) and the [memory adoption playbook](memory-adoption-playbook.md). There is no test harness that automates or verifies the gate &mdash; promotion is a manual judgement call today.
- **Router-hints loop.** `summon-memory.ps1` emits the header (the `.sh` version does not). There is no measurement loop that confirms Copilot's auto-router actually changes its model choice because of it. The numbers in [copilot-auto-mode.md](copilot-auto-mode.md) are observations, not benchmarks.
- **Team-memory workflow.** `team-memory/inbox/` and `team-memory/canonical/` are empty. Approval gates are written down but no lesson has ever flowed through them. Single contributor today.
- **Cross-platform parity.** PowerShell scripts are the daily-driven path. Core `.sh` counterparts exist for: `loop`, `sync-memory`, `capture-observation`, `learn-memory`, `summon-memory`, `auto-capture-observations`, `lint-memory`, `run-weekly-memory`, `synthesize-observations`, and `_personal-root`. Missing `.sh` equivalents: `prune-observations`, `repair-mojibake`, `install-scheduled-task` (Windows-only, no `cron`/`launchd` installer ships). Shell scripts are marked executable but less exercised than the PowerShell path.
- **Skills layer (beyond anti-hallucination).** `skills/general/pr-review/`, `skills/salesforce/`, `skills/mulesoft/` each contain a single-file example. They serve as references for the skill format but have no test harness, no install path, and no evidence of consumption by any agent integration. Not decorative &mdash; but not shipped either.
- **Connectors layer.** `connectors/` contains only `README.md` and `template.md`. No connector contracts have been written; the directory is scaffolding for a future wave.

## Things the docs deliberately do NOT claim

- That this is a framework. The word is reserved for Wave 5 onward.
- That it has users beyond me.
- That the token-savings numbers in [copilot-auto-mode.md](copilot-auto-mode.md) are measured benchmarks. They're observed ranges with a documented measurement protocol you can run yourself.
- That `team-memory/canonical/` is canonical anything. It's an empty folder with an aspirational name.
- That the quality gate is a production-ready MCP server. The scorer works locally (100/100/100) and is integrated into every write path, but there's no MCP server or standalone deployment yet. That's Wave 5-A.
- That we compete with mem0 / Letta / Zep on storage. We don't store memories — we filter them.

If you find a claim on the site that contradicts this page, the site is wrong. Open an issue.
